Part 1 · Foundations

What is a Data Model?

Definition

A collection of conceptual tools for describing data, data relationships, data semantics, and consistency constraints.

💡 Analogy

A data model is like an architect's blueprint. It defines how the building (database) will be structured before a single brick (data) is laid.

Categories of Data Models

  • High-Level (Conceptual): Close to human perception (e.g., Entity-Relationship Model).
  • Implementation (Representational): Used in commercial DBMS (e.g., Relational Model, Object-Oriented).
  • Low-Level (Physical): Describes how data is actually stored on disk (e.g., B+ Trees, hashing).

Part 2 · Architecture

Three-Schema Architecture

Goal: To separate user applications from the physical database. Achieved via three levels of abstraction:

  • Internal Level: Physical storage structures, indexes, file organization.
  • Conceptual Level: Logical structure of the entire database (entities, data types, relationships). Hides physical details.
  • External Level (Views): Customized views for specific user groups. Hides parts of the conceptual schema.

Abstraction Layers

3 External Schema (User Views) ◀ Customized
2 Conceptual Schema (Community View) ◀ Logical
1 Internal Schema (Physical Storage) ◀ Hardware
Mappings ensure changes in one level don't break others

Part 2 · Architecture

Data Independence

Logical Data Independence

The capacity to change the conceptual schema without having to change external schemas or application programs.

✓ Add a new field ✓ Add a new entity ✗ Hard to achieve

Physical Data Independence

The capacity to change the internal schema without having to change the conceptual schema.

✓ Change storage devices ✓ Create new indexes ✓ Change file organization

Part 3 · Languages

Database Languages (SQL Sublanguages)

🏗

DDL (Data Definition)

Defines the database structure/schema.
Commands: CREATE, ALTER, DROP, TRUNCATE

📝

DML (Data Manipulation)

Manages data within schema objects.
Commands: SELECT, INSERT, UPDATE, DELETE

🔒

DCL (Data Control)

Controls access to data in the database.
Commands: GRANT, REVOKE

🔄

TCL (Transaction Control)

Manages changes made by DML statements.
Commands: COMMIT, ROLLBACK, SAVEPOINT

Part 3 · Interfaces

Database Interfaces

Types of Interfaces

  • Menu-Based (Forms): For browsing/standalone users (e.g., ATM screens).
  • Graphical (GUI): Visual representation of schema (e.g., MySQL Workbench).
  • Natural Language: Accepts English-like queries (requires complex parsing).
  • Parametric:** Restricted interfaces for bank tellers or reservation agents.

DBMS Interfaces in Code

  • Application Program Interface (API): ODBC, JDBC, PDO.
  • Embedded SQL: SQL statements embedded in host languages (C, Java, Python).
  • Dynamic SQL: SQL queries constructed and executed at runtime.
Modern apps rarely use raw SQL directly; they use ORMs (Object-Relational Mappers) like Prisma or Hibernate.

Part 4 · Environment

Database System Environment

🖥 Hardware

Secondary storage (HDD/SSD), primary memory (RAM), processors, and networking equipment.

💻 Software

The DBMS software itself, the Operating System, network software, and application programs.

👥 People

DBA, Database Designers, Application Programmers, and End Users (Naïve to Sophisticated).

📊 Data

The actual operational data and metadata (data dictionary) that acts as the bridge between components.

📜 Procedures

Instructions and rules governing the design, use, and maintenance of the database (e.g., backup routines).

Part 4 · Architecture

Centralized DBMS Architecture

How it Works

A single computer system (mainframe or powerful server) houses the entire DBMS, application programs, and user interfaces. Users access it via "dumb terminals" that only provide display and keyboard input.

✓ Simple to manage ✓ High security (centralized) ✗ Single point of failure ✗ Poor scalability ✗ High network latency for remote users

Part 5 · Client-Server

Two-Tier vs. Three-Tier Architecture

Two-Tier (Fat Client)

  • Tier 1 (Client): UI + Application Logic + DB API.
  • Tier 2 (Server): Database Engine.
  • Client connects directly to the DB.
✓ Faster for small LANs ✗ Security risk (DB credentials on client) ✗ High maintenance (update every client)

Three-Tier (Thin Client) ★

  • Tier 1 (Client): UI only (Browser/Mobile App).
  • Tier 2 (App Server): Business logic & processing.
  • Tier 3 (DB Server): Data storage & retrieval.
✓ Highly secure ✓ Scalable & easy to maintain ✓ Standard for modern web apps

Part 6 · Classification

Classification of Database Systems

By Data Model

  • Relational (SQL)
  • Object-Oriented (OODBMS)
  • NoSQL (Document, Key-Value, Graph)
  • Hierarchical & Network (Legacy)

By Number of Users

  • Single-user: Desktop DB (e.g., MS Access)
  • Multi-user: Enterprise DB (e.g., Oracle, PostgreSQL)

By Site Distribution

  • Centralized: Single location
  • Distributed: Data spread across multiple networked sites
  • Cloud: DBaaS (Database as a Service)

By Purpose

  • OLTP: Online Transaction Processing (Day-to-day ops)
  • OLAP: Online Analytical Processing (Data warehousing, BI)
