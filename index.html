Data Models & Instances vs Schemas

Part 1 · Data Models

Types of Data Models

What is a Data Model? A data model is a collection of conceptual tools for describing data, data relationships, data semantics, and consistency constraints. It defines how data is organized, stored, and manipulated in a database system.

Traditional Models

  • Hierarchical Model: Organizes data in a tree-like structure where each record has one parent and multiple children. Data is accessed through parent-child relationships.
    Example: File system directories, organization chart (CEO → Managers → Employees)
  • Network Model: Extends hierarchical model by allowing many-to-many relationships. Records can have multiple parents, forming a graph structure.
    Example: Social network where users can have multiple friends
  • Entity-Relationship (ER) Model: High-level conceptual model using entities (objects), attributes (properties), and relationships between entities. Used for database design.
    Example: Student entity enrolls in Course entity with Enrollment relationship

Modern Models

  • Relational Model: Most widely used model. Organizes data into tables (relations) with rows (tuples) and columns (attributes). Uses SQL for queries.
    Example: MySQL, PostgreSQL, Oracle, SQL Server
  • Object-Oriented Model: Represents data as objects (like in OOP programming). Supports inheritance, encapsulation, and complex data types.
    Example: ObjectDB, db4o
  • Object-Relational Model: Combines relational and object-oriented features. Supports complex data types and user-defined functions.
    Example: PostgreSQL with JSON support, Oracle with object extensions
💡 Real-World Application: A university database uses the Relational Model — Students table stores student info, Courses table stores course details, and Enrollment table connects students to courses they're taking.

Part 1 · Core Concepts

Schema vs Instance

Understanding Database Structure: A database has two important aspects — its design (schema) and its current data (instance). The schema defines the structure, while the instance contains the actual data at any given moment.

📋 Database Schema (Design)

Definition: The overall logical structure and design of the database. It specifies what data will be stored, the data types, constraints, and relationships between tables.

  • Defines table structure (columns, data types)
  • Specifies constraints (PRIMARY KEY, FOREIGN KEY)
  • Changes rarely (only when design needs modification)
  • Created by Database Administrator (DBA) or designer
  • Independent of actual data
SQL Example:
CREATE TABLE Student (
  ID INT PRIMARY KEY,
  Name VARCHAR(50),
  Age INT,
  Department VARCHAR(30)
);

📊 Database Instance (Data)

Definition: The actual data stored in the database at a specific moment in time. Also called the "state" of the database. Changes with every data operation.

  • Contains actual records/rows of data
  • Changes frequently with INSERT/UPDATE/DELETE
  • Represents a snapshot of the database
  • Different at different points in time
  • Must conform to the schema constraints
Current Data (Instance):
(101, 'Ram Sharma', 20, 'CSIT')
(102, 'Sita Thapa', 19, 'BCA')
(103, 'Hari Prasad', 21, 'BIM')
🎯 Simple Analogy: Schema = Blank form template with fields | Instance = The filled form with actual student information

3-Schema Architecture & Data Independence

Part 2 · Architecture

Three-Schema Architecture

Purpose: The three-schema architecture separates user applications from the physical database. It provides data abstraction by hiding complex storage details from users, making the system easier to use and maintain.

3

View Level (External Schema)

What users see: Each user group sees only the data relevant to them. Multiple external views can exist for the same database, hiding sensitive or unnecessary information.

Example: A student sees only their own grades and attendance. A teacher sees all students in their class. The HR department sees employee salaries. Each has a customized view of the same database.
2

Logical Level (Conceptual Schema)

What data is stored: Describes the entire database structure — what entities exist, their attributes, relationships between them, and data types. Hides physical storage details.

Example: Defines Student table (ID, Name, Age, Department), Course table (CourseID, Title, Credits), and Enrollment relationship connecting students to courses.
1

Physical Level (Internal Schema)

How data is stored: Describes low-level storage structures — file organization, indexing methods, data compression, encryption, and access paths. Most complex level.

Example: Student records stored in B+ tree index on ID field, data compressed using gzip, stored on SSD with 4KB block size, hash index on Department for fast lookup.

Part 2 · Independence

Database Languages & Commit vs Rollback

Part 3 · Languages

Database Languages (SQL Sublanguages)

Database Languages: DBMS provides specialized languages for different database operations. SQL (Structured Query Language) is divided into four sublanguages, each serving a specific purpose in database management.

🏗️

DDL

Data Definition Language

Defines and modifies database structure (schema). Used to create, alter, and drop database objects like tables, indexes, and views.

CREATE TABLE
ALTER TABLE
DROP TABLE
CREATE INDEX
TRUNCATE TABLE
📝

DML

Data Manipulation Language

Manages data within database objects. Used to retrieve, insert, update, and delete actual data records.

SELECT (retrieve)
INSERT (add)
UPDATE (modify)
DELETE (remove)
🔒

DCL

Data Control Language

Controls access to data and permissions. Used to grant or revoke privileges to users and manage security.

GRANT (give access)
REVOKE (remove access)
🔄

TCL

Transaction Control Language

Manages transactions and ensures data consistency. Controls changes made by DML statements and handles recovery.

COMMIT (save)
ROLLBACK (undo)
SAVEPOINT (checkpoint)

Part 3 · Transactions

COMMIT vs ROLLBACK

Transaction Management: A transaction is a sequence of operations performed as a single logical unit of work. COMMIT and ROLLBACK are TCL commands that control whether changes are permanently saved or discarded.

✅ COMMIT Command

Purpose: Permanently saves all changes made during the current transaction to the database.

  • Makes all changes permanent and visible to other users
  • Cannot be undone after execution
  • Writes changes to disk (durability)
  • Releases locks held by the transaction
  • Ends the current transaction
Example - Bank Transfer:
BEGIN TRANSACTION;
UPDATE Account SET balance = balance - 1000 WHERE id = 1;
UPDATE Account SET balance = balance + 1000 WHERE id = 2;
COMMIT; ← Money transferred permanently

↩️ ROLLBACK Command

Purpose: Undoes all changes made during the current transaction, restoring the database to its previous state.

  • Cancels all uncommitted changes
  • Restores database to last COMMIT point
  • Used when errors occur or logic fails
  • Maintains data consistency (ACID property)
  • Releases locks and ends transaction
Example - Error Handling:
BEGIN TRANSACTION;
UPDATE Account SET balance = balance - 1000 WHERE id = 1;
-- Error: Account 2 doesn't exist!
ROLLBACK; ← Money returned to Account 1
💡 Real-World Scenario: When you transfer money online, the system: (1) Deducts from your account, (2) Adds to recipient's account. If step 2 fails (network error, invalid account), ROLLBACK ensures your money isn't lost. Only COMMIT after both steps succeed.

DB System Environment & Structure

Part 4 · Environment

Database System Environment

Complete Database Ecosystem: A database system is not just software — it's a complete environment consisting of five major components that work together to store, manage, and provide access to data.

🖥️

Hardware

Physical devices required to store and process data. Includes secondary storage (HDD/SSD for data files), primary memory (RAM for caching), processors (CPU for query execution), and network devices for client-server communication.

💻

Software

Programs that manage the database. Includes the DBMS itself (MySQL, PostgreSQL, Oracle), the operating system (Windows, Linux), application programs that access the database, and network software for distributed systems.

People

Users who interact with the database system. Includes Database Administrators (DBA) who manage the system, Database Designers who create the schema, Application Programmers who write code, and End Users who query the data.

Data

The most important component — the actual information stored. Includes operational data (student records, transactions) and metadata (data about data, stored in data dictionary/catalog describing structure and constraints).

📜

Procedures

Instructions and rules for using the database system. Includes installation procedures, backup and recovery routines, security policies, user manuals, and guidelines for database design and maintenance.

Part 4 · Structure

Database System Structure

Internal Components: The DBMS has two major functional components that work together — the Storage Manager handles low-level data storage and retrieval, while the Query Processor translates and executes high-level queries from users.

⚙️ Storage Manager

Role: Acts as an interface between low-level data stored in files and application programs. Responsible for storing, retrieving, and updating data on disk while ensuring efficiency and consistency.

Key Components:

  • Authorization & Integrity Manager: Checks user permissions and enforces constraints (PRIMARY KEY, FOREIGN KEY) before allowing operations
  • Transaction Manager: Ensures database remains consistent despite system failures. Implements ACID properties through locking and logging
  • File Manager: Manages disk space allocation, organizes data into files and blocks, handles data structures like B+ trees
  • Buffer Manager: Caches frequently accessed data in RAM (buffer pool) to reduce disk I/O and improve performance
Example: When you run SELECT * FROM Student, the buffer manager checks if Student data is in RAM. If not, file manager reads it from disk into buffer for fast access.

🔍 Query Processor

Role: Translates high-level SQL queries into low-level instructions that the storage manager can execute. Optimizes queries for best performance.

Key Components:

  • DDL Interpreter: Processes DDL statements (CREATE, ALTER, DROP) and records metadata in the data dictionary/catalog
  • DML Compiler: Translates DML queries (SELECT, INSERT) into an evaluation plan — a sequence of low-level operations. Includes query optimization to choose the fastest execution method
  • Query Evaluation Engine: Executes the low-level instructions generated by the DML compiler, coordinating with storage manager to fetch/modify data
Example: SELECT * FROM Student WHERE Age > 20 → DML Compiler creates plan: "Use Age index → Fetch matching records" → Evaluation Engine executes the plan

Types of Database Systems

Part 5 · System Types (1/2)

Centralized & Personal Computer Databases

Single-Location Systems: These database architectures keep all data and processing in one place. They are simpler to manage but have limitations in scalability and fault tolerance.

🏢 Centralized Database

Architecture: All data is stored, maintained, and accessed from a single physical location (mainframe or powerful central server). Users connect via terminals or a local network.

Example: A university's central database server where all student records, faculty data, and course information are stored. All campus departments access this single server.
✓ Easy to manage, maintain, and secure ✓ No data redundancy or synchronization issues ✗ Single point of failure (if server dies, everything stops) ✗ Poor scalability and potential network bottlenecks

💻 Personal Computer (PC) Database

Architecture: The entire DBMS, application software, and data files reside on a single personal computer. Designed specifically for single-user access with limited data volume.

Example: An MS Access database on your laptop to track personal expenses, or a local SQLite database for a small desktop inventory application.
✓ Very simple, inexpensive, and easy to install ✓ No network connection required to function ✗ Strictly single-user (no concurrent access) ✗ Limited data capacity and processing power

Part 5 · System Types (2/2)

Client/Server & Distributed Databases

Distributed Processing Systems: These architectures split the workload across multiple machines, offering better scalability, performance, and reliability for modern, large-scale applications.

🌐 Client/Server Database

Architecture: Processing is distributed. Client machines handle the user interface and application logic, while the central server manages data storage, query processing, and transaction management.

Example: An e-commerce website where your browser (client) sends requests to a web server, which then queries a MySQL database server to fetch product info and process your order.
✓ Highly scalable and better security (credentials stay on server) ✓ Centralized data management with distributed processing ✗ Heavy network dependency ✗ Server can become a bottleneck under heavy load

🔗 Distributed Database (DDBMS)

Architecture: A single logical database is physically fragmented and stored across multiple computers at different geographic locations, connected via a network. Appears as one system to the user.

Example: A global bank with branches in Kathmandu, Pokhara, and Butwal. Each branch has a local database, but all are synchronized. A customer can access their account from any branch.
✓ High availability, reliability, and modular growth ✓ Better performance due to local data access ✗ Very complex data synchronization and concurrency control ✗ Expensive infrastructure and security challenges

DBMS Classification & Chapter Summary

Part 6 · Classification

Classification of Database Management Systems

DBMS Categories: Database Management Systems can be classified in three main ways — by the data model they use, by the number of users they support, and by how the database is distributed across locations.

📊 Based on Data Model

Classification by how data is organized and structured:

  • Relational DBMS (RDBMS): Uses tables with rows and columns. Most popular today.
    Examples: MySQL, PostgreSQL, Oracle, SQL Server
  • Hierarchical DBMS: Tree structure with parent-child relationships. Legacy systems.
    Example: IBM IMS
  • Network DBMS: Graph structure allowing many-to-many. Legacy systems.
    Example: IDMS
  • Object-Oriented DBMS: Stores data as objects with methods.
    Example: ObjectDB
  • NoSQL DBMS: Non-relational, flexible schema for big data.
    Examples: MongoDB, Cassandra, Redis

👥 Based on User Numbers

Classification by concurrent user support:

  • Single-User DBMS:
    • Supports only one user at a time
    • Typically runs on personal computers
    • Used for individual, lightweight applications
    Examples: MS Access, SQLite (single-user mode)
  • Multi-User DBMS:
    • Supports multiple concurrent users
    • Provides advanced concurrency control & locking
    • Used in enterprise and web environments
    Examples: Oracle, MySQL, PostgreSQL, SQL Server

🌍 Based on Distribution

Classification by physical location of data:

  • Centralized DBMS:
    • All data stored at a single physical location
    • Single database server for all users
    • Easier to manage, backup, and secure
    Example: Small business with one local server
  • Distributed DBMS (DDBMS):
    • Data spread across multiple networked sites
    • Each site has local autonomy and control
    • Appears as a single logical database to users
    Example: Multinational corporation with global offices

Part 6 · Summary

Chapter 2 — Key Takeaways

Chapter Overview: This chapter covered the fundamental concepts of database systems, from data models and architecture to languages and system types. These concepts form the foundation for understanding how modern databases work.

1

Data Models: Define how data is structured. Relational model (tables) is most widely used today. Traditional models (hierarchical, network) are legacy systems.

2

Schema vs Instance: Schema is the database design/structure (rarely changes). Instance is the actual data at a moment in time (changes frequently with operations).

3

Three-Schema Architecture: Physical level (storage), Logical level (design), View level (user interface). Provides data abstraction and hides complexity.

4

Data Independence: Physical independence (change storage without affecting logic) is easier. Logical independence (change structure without affecting views) is harder.

5

Database Languages: DDL (structure), DML (data manipulation), DCL (security/permissions), TCL (transaction control). SQL combines all four.

6

Commit/Rollback: COMMIT permanently saves transaction changes. ROLLBACK undoes uncommitted changes. Essential for data consistency and error handling.

7

DB Components: Hardware, Software, People, Data, Procedures. Storage Manager handles disk I/O. Query Processor translates and executes SQL.

8

System Types: Centralized (single location), PC (single user), Client/Server (distributed processing), Distributed (multiple locations). Each serves different needs.
